Webhook Troubleshooting Runbook

Yesterday this worked because Twilio had a public URL that could reach the local Next app. With quick Cloudflare tunnels, that URL usually only works while the tunnel process is alive. If the tunnel stops or restarts, Twilio may keep calling an old URL.

  1. Confirm the app is running: App Home, Admin, Health API, and Local Twilio Webhook should show Up.
  2. Confirm the public tunnel is alive: Public Twilio Webhook should show Up. If it is down, start a new tunnel to http://localhost:3001.
  3. Copy the new tunnel URL into Twilio: use /api/webhooks/twilio for incoming messages and /api/webhooks/twilio/status for status callbacks, both with method POST.
  4. Click Save in Twilio: visible fields are not enough; the sandbox must save the values.
  5. Send an inbound WhatsApp message: reply from your phone to the Twilio sandbox number. This should create a POST /api/webhooks/twilio line in the Next server logs.
  6. Interpret failures: no POST means tunnel/Twilio forwarding; 401 means signature/token mismatch; 500 means app/database handling error.

Version Stability Policy

This project avoids latest-runtime drift. Local development is pinned to Node 22.x and npm 10.x, with framework/package upgrades handled deliberately instead of automatically chasing newest releases.

  • Runtime: Node 22.x only for this project.
  • Package manager: npm 10.x only for this project.
  • Enforcement: `.nvmrc`, `package.json` engines, `.npmrc` `engine-strict=true`, and runtime prechecks before dev/build/start/lint.
  • Upgrade rule: prefer a stable LTS or proven release line, not the newest major version on day one.

1) Core Stack

Frontend

Next.js

Admin UI, dashboards, testing screens, customer support tools, and internal management pages.

API Layer

Next.js API Routes / Backend Services

Receives webhook events, validates requests, orchestrates business logic, and returns responses.

AI Orchestration

LangChain

Builds prompt flows, context retrieval, tool calling, memory handling, and response generation.

Data Layer

Supabase

Stores customers, conversations, knowledge records, prompt configs, audit logs, and app settings.

Messaging

WhatsApp API

Inbound and outbound customer messaging via Meta Cloud API or a provider such as Twilio.

2) End-to-End Message Flow

1. Customer sends message

The customer sends a WhatsApp message to the business number.

→
2. WhatsApp webhook triggers

The provider sends the event payload to your backend webhook endpoint.

→
3. API layer processes request

Your backend authenticates, normalizes the message, identifies the customer, and starts the workflow.

→
4. LangChain builds context

Pull customer history, business rules, product information, and relevant knowledge from Supabase.

→
5. AI generates a response

The model drafts a context-aware reply, optionally invoking tools or workflows.

→
6. Response sent to customer

The backend stores the interaction and sends the response back through the WhatsApp API.

Inbound: WhatsApp → Webhook Processing: API + LangChain Persistence: Supabase Outbound: API → WhatsApp

5) Recommended Initial Setup Choices

  • Framework: Next.js
  • Router choice: Pages Router is acceptable for a simpler initial build, though App Router is better for long-term modernization.
  • Backend approach: Start with Next.js API routes for speed; split into separate services later only if scale requires it.
  • Database: Supabase PostgreSQL as the operational database.
  • AI framework: LangChain in the backend only, not as a visual setup option in the project wizard.
  • Messaging provider: Meta WhatsApp Cloud API or Twilio WhatsApp.
Practical note: LangChain is not normally selected in the project creation UI. You create the app first, then install LangChain as a package in the backend code.

6) Implementation Plan

Phase 1 — Foundation

  • Create the Next.js project.
  • Set up Supabase project and environment variables.
  • Create initial tables: customers, conversations, messages, knowledge_base, logs.
  • Set up local development, deployment, and secrets handling.

Phase 2 — WhatsApp Integration

  • Register the WhatsApp business channel.
  • Create webhook endpoints for inbound messages and delivery status updates.
  • Validate provider signatures and handle retries safely.
  • Store raw inbound events in Supabase for traceability.

Phase 3 — AI Orchestration

  • Install LangChain and connect the chosen LLM provider.
  • Build a response chain that uses business rules and customer context.
  • Add retrieval logic for FAQs, policies, and customer history.
  • Return structured replies to the API layer.

Phase 4 — Business Workflow Layer

  • Add intents such as greeting, support, sales inquiry, application status, and escalation.
  • Add human handoff for sensitive or failed conversations.
  • Create admin views for monitoring and intervention.
  • Track conversation outcomes and response quality.

Phase 5 — Hardening and Scale

  • Add rate limiting, retries, error queues, and observability.
  • Improve security, role-based access, and audit controls.
  • Optimize prompts, retrieval quality, and response timing.
  • Expand to more workflows and automation use cases.

7) Core Decision Points

Use Pages Router now?

Yes, for a fast start.

Low friction for early build-out if that is what you are already using.

Use LangChain in frontend?

No.

Keep it in server-side API logic so keys and workflows remain secure.

Need NoSQL?

Not necessarily.

Supabase is PostgreSQL and is usually enough to start, even for conversational systems.

Split microservices immediately?

No.

Start monolithic, then split when scale, latency, or team boundaries justify it.
