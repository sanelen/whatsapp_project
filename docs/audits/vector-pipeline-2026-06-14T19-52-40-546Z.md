## Findings

1. [HIGH] Real file uploads are not implemented end-to-end - Multipart uploads fail because `/api/kb/upload` expects JSON and does not parse uploaded files, so the current app does not have a working document upload pipeline.
2. [HIGH] Binary/document file types are accepted without real parsing - PDF, DOCX, XLSX, and EXE-style fixtures can be indexed as plain text payloads, which means the API currently treats caller-provided strings as already-parsed content instead of validating or extracting real file contents.
3. [HIGH] Supabase Storage is not part of the current file path - The current codebase does not upload files into Supabase Storage or map organization/property folders, so storage lifecycle and cleanup are unimplemented.
4. [MEDIUM] Retrieval is not property-scoped in the live API - The search RPC filters by `sourceType` only, so cross-property knowledge bleed is still possible until property/chatbot scoping is added to the API and metadata filters.
5. [MEDIUM] Chat route depends on internal HTTP fetch for knowledge search - The chat route calls `/api/kb/search` over HTTP using `APP_URL`, which adds coupling to server hostname/auth behavior and can mask auth issues during local bypass mode.
